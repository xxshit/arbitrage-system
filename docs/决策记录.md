# 决策记录

## 2026-07-13：先构建本地可运行版本

- 决策：先实现本地 Web 看板和策略管理，不直接接入真实下单。
- 原因：先验证套利信息展示、策略流程与数据库结构，避免在没有凭证和风控规则时产生交易风险。

## 2026-07-13：数据库采用 MySQL 优先、SQLite 回退

- 决策：通过 `DATABASE_URL` 使用 MySQL；缺省时使用 SQLite。
- 原因：用户需要 MySQL，但本地首启不应被数据库尚未安装或未配置阻断。

## 2026-07-13：行情使用模拟数据

- 决策：当前用模拟行情驱动看板。
- 原因：尚未提供交易所 API 凭证；真实行情接入前需要明确交易所范围、访问频率与数据校验方案。

## 2026-07-13：市场数据采用滚动缓存而非永久留存

- 决策：最新行情覆盖写入数据库；资金费率历史仅保存最近 30 天。
- 原因：排序与翻页不应触发交易所请求，同时避免数据库因高频报价或不可变历史资费无限增长。
- 约束：新增需要永久保存的数据集前，应先与用户确认留存目的和期限。

## 2026-07-16：Lark 日报使用单列灰底富文本卡片

- 决策：日报趋势推送使用 Lark Card JSON 2.0 的单列灰底容器，不再使用纯文本消息。
- 原因：纯文本消息无法让同一条内容中的不同字段拥有独立颜色；富文本卡片支持按片段设置自定义 RGBA 色值。
- 表达约束：状态标识统一使用圆点 `●`，仅以不同色阶区分强弱、涨跌与待观察状态；不混用方块、圆形等不同图形。

## 2026-07-16：Lark 日报三币合并与指标当前值

- 决策：Lark 趋势日报的共振前三个币合并为同一条灰底卡片消息，不再拆成三条独立消息。
- 排版约束：同一条 Lark 消息内保留三个独立灰底块，每个币一个块；不额外增加总标题、分隔线或强制窄卡片，避免消息体被压窄导致高度过高。
- 指标口径：价格、持仓、多空人数比的颜色只表达数值涨跌；上涨为绿色，下跌为红色，不能把“策略上有利”误显示为绿色。
- 数据留存：持仓与多空人数比的当前值继续写入数据库和日报接口；Lark 推送按用户截图版保持紧凑，只显示变化量。

## 2026-07-16：Lark 走势分析推送独立通道

- 决策：每日 08:00 的趋势日报继续使用 `LARK_DAILY_TREND_WEBHOOK`，只推送当日排序最靠前的 3 个共振币。
- 决策：新增 `LARK_THOUGHT_ANALYSIS_WEBHOOK`，专门用于正在分析的币种走势盯盘，不与日报机器人混用。
- 触发约束：走势分析机器人不做全市场广播；只监控“思路分析”中正在跟踪的币。只有近 30M、1H、2H 三个窗口全部出现价格、持仓、CVD 同向共振，且价格变化至少 0.8%、持仓变化至少 1.0%，才推送一次。
- 去重约束：同一币种同一信号键 6 小时内不重复推送；后续只有突破/跌破关键位或信号类型变化才再次推送。
- 复盘约束：若走势分析推送后被市场证伪，必须在趋势研判日志保留原判断、写明错误原因，并把缺陷沉淀成新的过滤规则。

## 2026-07-16：日报推送必须持久化去重

- 问题：日报 08:00 推送原先依赖内存变量判断当天是否已推送；本地网站重启后内存清空，导致 8 点后每次重启都有可能重复推送日报。
- 决策：日报推送成功后写入 `lark_push_state`，使用 `channel=daily_trend`、`symbol=ALL`、`signal_key=YYYY-MM-DD` 做持久化去重。
- 约束：日报推送不得只依赖进程内变量；重启网站不能导致当天日报重复推送。日报只允许在北京时间 08 点这一小时触发，超过 09 点不补发。

## 2026-07-17：AKE 思路盯盘不消耗模型 token

- 决策：AKE 思路盯盘由本地后端每 5 分钟请求公开行情接口并计算，不调用模型；页面打开时也只从本地接口读取结果。
- 触发约束：走势机器人只在“犄型走势”确认时推送，即近 30M、1H、2H 同时出现价格转强、持仓增加、多空人数比下降、CVD 上涨。没有确认时只更新页面，不推送。
- 安全约束：Lark webhook 不写入 HTML，不暴露给浏览器；所有推送由后端读取 `.env` 并执行。
