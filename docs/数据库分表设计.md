# 数据库分表设计

目标：交易系统里的数据不能混在一起。实时行情、历史行情、资费、报警、日报、思路复盘、交易所元数据都要按功能分表，方便查询、维护、备份和以后迁移云服务器。

## 1. 实时行情快照表

这类表只保存“最新状态”，页面排序、分页、筛选都优先读这些表，不要因为用户点击页面就重新请求交易所。

| 表名 | 功能 |
| --- | --- |
| `latest_market_snapshot` | 现多期空：现货交易所对 Binance 合约的最新价差、基差、资费、成交量、持仓量 |
| `latest_dual_futures_snapshot` | 期多期空：BY/OK/BN 合约路径的最新开差、平差、基差、资费差、持仓量、交易量 |

## 2. 历史行情与资费表

这类表用于趋势灯、涨跌幅、详情弹窗、日报趋势和历史资费统计。

| 表名 | 功能 |
| --- | --- |
| `futures_price_history` | Binance 合约价格历史采样 |
| `dual_futures_price_history` | BY/OK/BN 合约价格历史采样 |
| `funding_rate_record` | Binance 历史资金费率，滚动保留最近 30 天 |

规则：历史资费是固定事实，写入后优先复用；超过保留周期再清理，避免 MySQL 越来越大。

## 3. 报警与基差打开表

这类表用于报警中心、全局 TIPS、基差打开次数、最大值和误报警复查。

| 表名 | 功能 |
| --- | --- |
| `alert_event` | 报警事件，包括开差、基差、速变等 |
| `basis_tracking` | 基差第一次打开、方向、最近最大值、最大值时间 |
| `basis_expansion_log` | 基差每扩大 0.2% 的打开记录 |

规则：同一币种同一路径的报警不能无脑重复；只有第一次触发，或绝对值比上一轮报警扩大至少 0.2%，才刷新报警。

## 4. 交易所低频元数据表

这类数据通常不频繁变化，拉到后存 MySQL，以后页面直接读库。

| 表名 | 功能 |
| --- | --- |
| `index_component_snapshot` | BN/BY/OK 合约指数成分 |
| `transfer_network_snapshot` | 现货交易所充提链与充提状态 |
| `listing_state` | 上下架状态基线 |
| `listing_event` | 上架、下架、公告事件，包含公告时间和具体生效时间 |

## 5. 日报、趋势与推送状态表

这类表用于每天固定时间的趋势扫描、Lark 推送去重和任务健康检查。

| 表名 | 功能 |
| --- | --- |
| `daily_horn_signal` | 每日犄型走势候选，分 30M 和 4H |
| `lark_push_state` | Lark 推送去重，避免重复发同一个信号 |
| `automation_status` | 自动任务最近开始、成功、失败时间 |

## 6. 策略与后续要新增的思路复盘表

已有：

| 表名 | 功能 |
| --- | --- |
| `strategy` | 基础策略配置 |

下一步建议新增，不要再把思路分析写死在代码里：

| 建议表名 | 功能 |
| --- | --- |
| `trade_thesis` | 单个币的交易思路：币种、方向、入场价、入场时间、用户观点、Codex 观点、当前状态 |
| `trade_thesis_snapshot` | 盯盘快照：价格、资费、基差、持仓、CVD、量能、判断 |
| `trade_thesis_review` | 复盘：哪里对、哪里错、止盈止损是否合理、后续修正 |

这样 AKE、T、US 这类分析就会变成可追踪、可统计胜率、可复盘的数据，而不是零散文本。

## 当前结论

当前 MySQL 已经不是“所有数据混在一张表里”，核心模块已经分表。下一步最需要拆出来的是“思路分析/盯盘/复盘”三张表，因为这是后期你和我共同进化交易思维的核心资产。
