# 架构设计

## 5 秒行情刷新分层

- 盘口、标记价、指数价与当前资金费率属于套利关键路径，后台独立任务以 5 秒为目标周期拉取并先写入最新快照。
- 交易规则、合约列表等元数据按 1 小时缓存；资金费率周期按 60 秒缓存；24 小时成交统计按 20 秒缓存。
- 持仓量与 OKX 单合约资金费率属于辅助信息，采用小批次、1 秒超时的补充同步；请求慢或失败时沿用最近有效缓存，不能阻塞开差与基差快照。
- 页面只读取 MySQL 最新快照；排序、分页与筛选不向交易所发请求。刷新异常时保留上一次有效展示。

## 报警有效期与日报

- 报警中心保留确认后的历史记录；全局 TIPS 仅显示创建或最后一次扩展后 120 秒内仍有效的全部报警，不设三条上限。
- 同一币种、同一策略报警仅在开差或基差绝对值再扩大至少 0.2% 时覆盖更新时间，因此重新获得完整 120 秒有效期并再次触发声音。
- 日报趋势从本地 Binance 合约价格历史读取 24H、7D 涨跌幅；Binance USDT 永续合约的首次元数据快照建立基线，其后检测到新合约或合约消失时写入本地上下架事件表。

## 当前版本

ArbiScope 是一个本地运行的 Python Web 应用，用于展示数字资产套利机会和管理监控策略。

```text
浏览器页面
    │ HTTP / JSON
Flask 应用（app.py）
    ├── 套利机会计算（当前为模拟行情）
    ├── 策略管理 API
    └── SQLAlchemy 数据访问层
             │
      MySQL（生产建议）/ SQLite（本地默认）
```

## 技术组成

- 后端：Python、Flask、Flask-SQLAlchemy
- 数据库：MySQL + PyMySQL；若未设置 `DATABASE_URL`，自动使用本地 SQLite
- 前端：原生 HTML、CSS、JavaScript
- 配置：`.env`（由 `.env.example` 复制生成）

## 数据库

当前包含 `strategy` 表：

| 字段 | 含义 |
| --- | --- |
| id | 策略编号 |
| name | 策略名称 |
| mode | 套利模式 |
| symbol | 交易对 |
| enabled | 启用状态 |
| created_at | 创建时间 |

## 后续数据流

1. 各交易所适配器拉取现货、永续合约价格和资金费率。
2. 标准化交易对、精度、手续费与合约面值。
3. 套利引擎按策略计算可执行净收益。
4. 风控引擎过滤额度、滑点、流动性、资金费率结算时间和账户余额。
5. 合格机会写入数据库并推送至看板；真实下单必须独立于监控模块。

## 行情缓存与留存

- 交易所公开行情由后台任务以 5 秒为目标节拍拉取一次，写入 `latest_market_snapshot`；单轮请求若超过 5 秒，完成后立即进入下一轮，不再额外等待。页面排序、翻页、筛选仅从数据库读取，不会因点击动作再次请求交易所。
- 实时行情、历史资费同步与价格历史回补使用独立后台任务；历史写入不会阻塞 5 秒行情快照与报警判断。
- 本地 SQLite 下，历史资费同步与 K 线回补按低优先级、错峰运行；云端 MySQL 可按资源情况提高历史同步频率。实时行情和报警始终优先于历史任务。
- `funding_rate_record` 保存 Binance 历史资金费率，后台按批次回填；历史费率不可变，页面直接读取数据库。
- 资金费率记录采用滚动保留策略：仅保留最近 30 天，超过 30 天自动删除；最新行情快照按“交易对 + 现货交易所”覆盖更新，不累计增长。
- 两个套利页面均预留币种搜索入口；现多期空搜索由数据库在排序、分页前筛选，支持 `BTC`、`BTC/USDT` 等输入且不触发交易所请求。期多期空页面已预留同样的搜索状态，待该行情模块接入后复用。

### 期多期空行情模块

- `LatestDualFuturesSnapshot` 将 Bybit、OKX、Binance 的公开永续合约盘口聚合后保存；页面通过 `/api/dual-futures` 从本地数据库读取、筛选、全量排序、再分页，不会因搜索或点击排序而直接请求交易所。
- 首批聚合路径为 `Bybit → Binance`、`OKX → Binance`、`Bybit → OKX`。合约路径两端分别保存盘口、基差、资金费率与结算信息，资金费差在服务端按空端减多端计算。
- `DualFuturesPriceHistory` 按交易所与币种存储 5 分钟价格快照，用于多端、空端各自的趋势灯与涨跌幅。新接入币种会先积累实时快照，历史补齐可沿用后台低优先级任务逐步扩展。
- OKX 当前资金费率需要按合约读取，系统使用分批、5 分钟缓存刷新以控制请求频率；尚未轮到的合约显示为待同步，而不会用错误的默认费率代替。
- 跨所同代码但不同标的会形成伪价差。聚合时通过两端中间价一致性阈值过滤（相差超过 50% 不展示），避免将代码碰撞误报为套利或拉升机会。
- 现多期空的任一排序（基差、各周期资费差、开差、平差）均先覆盖全部符合条件的币种，完成全量排序后再按每页 30 个币种分页；页面支持上一页、下一页及直接选择目标页。

### 趋势日报：犄型走势扫描

- 每日北京时间 08:00，系统先基于本地 Binance 价格快照筛出上涨候选，再按需读取候选币种的 Binance 合约 OI 历史、全市场账户多空比与 K 线主动成交额，避免为全市场高频计算 CVD。
- 候选条件：30 分钟观察最近 50 根已收线（约 25 小时），4 小时观察最近 25 根已收线（约 4 天）；两种口径均要求价格上涨、OI 上涨、账户多空比下降，CVD 净主动买入为“确认上涨”。每日按 0-100 结构置信分取前 20：三项累计强度与逐根一致性各占权重，CVD 上涨给予确认加分，不按涨幅排名。
- 该扫描是研究筛选器，不是交易信号；OI 与账户比的口径限制、CVD 背离与后续验证规则记录在 `趋势研判日志.md`。
- 切换现多期空的排序字段或正反方向时，页面自动回到第 1 页；“最近更新”旁展示基于最新快照计算的下一次后台行情同步倒计时。现多期空已接入 Binance、Gate、Bitget 公开 API，总览看板保留为演示数据。
- 价格趋势使用 Binance 合约中间价，每 5 分钟采样一条并滚动保留 8 天；支持 5MIN、15MIN、30MIN、1H、12H、24H、3D、7D。币种名称上方以绿灯（上涨）、红灯（下跌）、灰灯（数据不足或持平）展示趋势；独立的纵向涨跌幅列展示数值，8 个周期均支持全量排序。
- 现多期空的当前排序方式在搜索框旁实时展示；报警中心与全局 TIPS 每 5 秒检查一次新报警。顶部可切换“报警声音：开启／静音”；声音仅在出现新的、更强报警时播放一次，避免轮询时重复提示。

## 持续运行部署

- 本地电脑关机、休眠或断网时，后台采集、报警与价格历史回补都会停止；本地任务计划只能在电脑再次开机后自动恢复，不能覆盖离线期间的实时报警。
- 推荐将 Flask 后台采集服务与 MySQL 部署到一台 24 小时在线的云服务器；网页可继续由本地浏览器访问，或通过安全域名访问云端。
- 云端服务需要设置开机自启、进程守护、数据库自动备份、日志轮转与 HTTPS；交易所 API 密钥仍应最小权限、禁止提现。
- 若暂不使用云服务器，可用长期通电联网的 NAS、小主机或旧电脑承担后台任务；关闭日常电脑不影响采集。
- 项目已提供 Docker、Compose 与 WSGI 启动框架，未来部署云端时无需改动业务逻辑；详见 `docs/云端部署准备.md`。
- 现多期空每个币种名称旁提供 Coinglass K 线跳转按钮，链接按 `https://www.coinglass.com/tv/zh/Binance_{交易对去除斜杠}` 生成，并在新标签页打开。
- 搜索框使用“前缀优先、包含回退”的联想规则：输入一个字母或汉字时优先展示以该内容开头的币种；若没有前缀结果，再展示包含该内容的模糊结果。中文输入只匹配交易所真实提供的中文币种名称，英文代码不使用中文译名联想；例如输入“币”可选“币安人生/USDT”，而 BNB 仅在输入英文时出现。

## 期多期空资费差自检约定

- 每次修改期多期空的结算、资费或排序计算后，必须在交付前对全部缓存路径自检，而非仅验证触发问题的单个币种。
- 自检至少覆盖：同周期路径必须标为“双方结算”；不同周期路径必须只计下一档先结算的一端；净资费数值必须满足“双方 = 空端资费 − 多端资费、仅多端 = −多端资费、仅空端 = 空端资费”。
- 若存在尚未同步的交易所资费，页面应明确显示待同步；不得填充默认费率、不得把缺失一端误算为双边结算。缓存补齐后再纳入全量数值核对。

### 指数成分缓存

- 期多期空的共指数据由独立后台任务分批读取 Binance `constituents`、Bybit `index-price-components` 与 OKX `index-components` 公开接口，写入 `index_component_snapshot`；页面和排序不会因鼠标悬浮或翻页而直接请求交易所。
- 指数成分属于低频基础资料：首次拉取后永久保存到数据库（本地 SQLite 与后续 MySQL 一致），页面始终从数据库读取；仅新出现的合约进入后台拉取队列。无指数或合约不存在时保存空成分结果并在界面明确标注，而不是以指数价格接近度伪造共指。
- 项目通用数据治理原则：实时行情、盘口、资金费率按各自刷新节奏采集；交易所元数据、指数成分、合约规格等低频且通常不变的数据首次入库后复用。若未来确认来源发生规则调整，再通过显式的维护任务更新，而不随页面操作重复请求。
- 自动化运行边界：行情拉取、缓存、数据库读写、倒计时与局部页面更新必须由网站自身完成。HTML 负责结构展示，浏览器 JavaScript 负责页面请求与局部渲染，Python 后端负责交易所采集、规则计算与 MySQL/SQLite 持久化；这些运行时流程不依赖对话模型，不消耗对话 Token。
